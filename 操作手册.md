# 格式转换工具 - 操作手册

## 📌 快速入门

本工具支持两种**输入模式**：
- **文件上传模式**：选择单个或多个文件进行转换
- **输入目录模式**：选择整个文件夹，自动处理文件夹内所有文件

### 系统要求

**最低配置**：
- 操作系统：Windows 10/11 (64位)
- 处理器：Intel i3 或同等性能处理器
- 内存：4GB RAM
- 硬盘空间：10GB 可用空间

**推荐配置**：
- 操作系统：Windows 10/11 (64位)
- 处理器：Intel i5 或更高
- 内存：8GB RAM 或以上
- 硬盘空间：50GB 或以上
- 显示器：1920×1080 或更高分辨率

**运行环境要求**：
- Java运行环境：JDK 8 或更高版本（已内置）
- Python：3.7+（用于OCR识别，已内置）
- Ghostscript：PDF处理依赖（已内置）

## 一、输入模式与规则选择

### 1. 两种输入模式对比

| 输入模式 | 适用场景 | 支持的规则 | 特点 |
|---------|---------|-----------|------|
| **文件上传模式** | 转换几个零散文件 | 规则0 | 手动选择文件，文件名作为PDF名 |
| **输入目录模式** | 批量转换整个文件夹 | 规则0 | 自动扫描文件夹，支持复杂目录结构 |

### 2. 如何选择输入模式？

```
📁 文件少（几个文件）
   └─→ 使用「文件上传模式」
   └─→ 点击"选择文件"按钮，选择要转换的文件

📂 文件多（整个文件夹）
   └─→ 使用「输入目录模式」
   └─→ 点击"输入目录"旁边的📁图标，选择文件夹
```

## 二、输出模式说明

| 输出模式 | 说明 | 适用场景 | 文字搜索 |
|---------|------|---------|---------|
| **双层PDF** | 图片层 + OCR文字层 | 📄 档案数字化（推荐） | ✓ 支持搜索复制 |
| **单层PDF** | 纯图片层（无文字） | 🖼️ 纯图像、防篡改 | ✗ 不能搜索 |
| **合并PDF** | 多个文件合并为一个PDF | 📑 案卷级整理 | 取决于单层/双层 |
| **拆分PDF** | 拆分多页文档 | ✂️ 提取页码或逐页拆分 | 取决于源文档 |

> ⚠️ **重要提示**：合并PDF功能只能在「输入目录模式」下使用！

### 模式详细说明

**双层PDF**（推荐用于档案数字化）
- 图片层：保留原始图像质量
- 文字层：通过OCR识别添加可搜索文字
- Office文档：保留原生文字 + OCR文字层

**单层PDF**（纯图片输出）
- 图片层：将内容转为纯图片
- 无文字层：不包含任何可搜索文字
- Office文档：先转PDF再转为纯图片，确保无文字层

**拆分PDF**（用于文档提取与单页化）
- 每一页都拆分成一个 PDF：将源文档的每一页提取为单独的 PDF 文件（输出文件名为 `文件名_001.pdf` 格式）。
- 提取指定页码范围：输入起始页码与结束页码，提取指定页面范围合并输出为一个 PDF 文件。
- 支持非PDF格式：如果输入是Office文档或图片，会自动先转为PDF再进行拆分。

## 三、文件夹规则详解

本系统仅保留输出文件规则 **规则0**。

### 规则0：不创建文件夹（默认）

- **说明**：所有转换后的文件直接输出到指定的输出目录中，不创建任何子文件夹。
- **示例**：
  ```
  输入：
    D:\输入\001.jpg
    D:\输入\002.jpg
  
  输出：
    D:\输出\001.pdf
    D:\输出\002.pdf
  ```
- **⚠️ 注意**：同名文件会互相覆盖！如果不同的子文件夹中包含同名文件，输出到同一个目录时会产生覆盖。系统在转换前会检测并警告。

### 6.4 基本操作流程

1. **选择输入模式**

2. **选择输入源**

   - 合并文件名（合并模式专用）

7. **开始转换**
   - 点击「开始转换」按钮
   - 查看进度条和当前处理文件
   - 等待转换完成

## 八、常见问题（FAQ）

### Q1：为什么我选了多个文件，只生成了一个PDF？
**A**：检查是否选择了「合并PDF」模式。合并模式会将所有文件合并成一个PDF。如需单独转换，请选择「双层PDF」或「单层PDF」。

### Q2：转换后的PDF文件名不对？
**A**：由于系统仅保留规则0，输出的文件名默认与原文件名保持一致。

### Q3：同名文件会覆盖吗？
**A**：由于只保留了规则0（不创建文件夹），所有转换后的文件均会直接输出到指定的根输出目录中。若存在同名源文件，它们转换后的PDF会发生覆盖风险。系统在开始转换前会自动检测同名冲突并给予警告提示，请注意修改文件名或分批转换。

### Q4：合并PDF功能无法使用？
**A**：合并PDF只能在「输入目录模式」下使用。请先点击「输入目录」选择文件夹，而不是选择文件。

### Q6：单层PDF和双层PDF有什么区别？
**A**：

| 对比项 | 单层PDF | 双层PDF |
|-------|--------|--------|
| **图片层** | ✓ 有 | ✓ 有 |
| **文字层** | ✗ 无（纯图片） | ✓ 有（可搜索） |
| **文件大小** | 较大 | 较小（带压缩） |
| **适用场景** | 需要纯图像、防篡改 | 需要搜索复制文字 |

**特殊说明 - Office文档转换**：
- **单层模式**：DOCX/DOC等Office文档先转为PDF，再转为纯图片，最终输出无文字层的纯图片PDF
- **双层模式**：保留Office原生的可搜索文字，并添加OCR识别的文字层

### Q7：DOCX转PDF后还能搜索文字吗？
**A**：取决于选择的输出模式：
- **单层PDF**：不能搜索，是纯图片格式
- **双层PDF**：可以搜索，保留了原生的可搜索文本

### Q8：程序启动后一直显示"正在连接服务器"？
**A**：可能的原因和解决方法：
1. **Java后端未启动**：等待30秒让后端完全启动
2. **端口被占用**：关闭其他占用8080端口的程序
3. **防火墙阻止**：允许程序通过防火墙
4. **重启程序**：完全关闭程序后重新启动

### Q9：转换速度很慢怎么办？
**A**：优化建议：
1. 减少同时转换的文件数量
2. 将输入文件复制到本地硬盘（非网络盘）
3. 关闭实时杀毒软件的文件扫描
4. 检查系统资源使用情况
5. 选择单层PDF模式（速度更快）

### Q10：OCR识别效果不理想？
**A**：改善方法：
1. 确保扫描图片清晰度≥300dpi
2. 调整图片对比度和亮度
3. 确保文字方向正确（不是倒置或倾斜）
4. 对于手写字体，识别效果可能较差

### Q11：合并PDF时文件顺序不对？
**A**：文件排序规则：
1. 合并模式支持按文件日期升序、日期降序、文件名称升序、文件名称降序排序
2. 文件名称排序使用自然排序，支持字符和数字混合（如 A1、A2、A10）
3. 建议使用规范的文件名（如001.jpg、002.jpg）

### Q12：如何处理包含大量文件的文件夹？
**A**：批量处理建议：
1. 先测试几个文件确认输出效果
2. 分批处理（每批500-1000个文件）
3. 使用统计功能核对文件数量
4. 定期备份已转换的文件

### Q13：程序占用内存过大？
**A**：解决方法：
1. 关闭程序后重新启动
2. 减少批量处理的文件数量
3. 选择单层PDF模式
4. 增加系统虚拟内存

### Q14：转换失败但没有错误提示？
**A**：排查步骤：
1. 查看日志文件 `app.log`
2. 检查输入文件是否损坏
3. 确认输出目录有写入权限
4. 检查磁盘空间是否充足
5. 尝试转换单个文件测试

### Q15：规则8什么时候使用？
**A**：规则8使用场景：
- **仅用于合并PDF模式**
- 适用于案卷级档案整理
- 当你需要将一个件号内的所有文件合并为一个PDF时
- 示例：将 `001/` 文件夹内的所有图片合并为 `001.pdf`

### 9.1 程序无法启动

**症状**：双击图标后程序没有反应

**解决方法**：
1. 检查是否已安装Java运行环境
2. 查看任务管理器是否有残留进程
3. 以管理员身份运行程序
4. 重新安装程序

### 9.2 转换中断或失败

**症状**：转换过程中程序停止或报错

**解决方法**：
1. 检查输入文件是否完整
2. 确认磁盘空间充足
3. 关闭其他占用资源的程序
4. 查看日志文件定位具体错误

### 9.3 OCR识别失败

**症状**：生成双层PDF时报错

**解决方法**：
1. 确认Python环境正常
2. 检查OCR模型文件是否完整
3. 尝试使用单层PDF模式
4. 更新显卡驱动（GPU加速需要）

### 9.4 文件名乱码

**症状**：转换后的文件名显示为乱码

**解决方法**：
1. 确保源文件名使用UTF-8编码
2. 避免使用特殊字符
3. 使用英文或数字命名
4. 检查系统区域设置

### 9.5 联系技术支持

如果以上方法无法解决问题，请联系技术支持：

**提供以下信息**：
